No retention policy on raw telemetry
CRITICAL
Cost

KQL database 'RawTelemetry' has no soft-delete or retention policy. At 12.5GB/day, storage will grow unbounded and costs will escalate.

→ Set retention policy: .alter table RawTelemetry policy retention softdelete = 90d. Archive older data to OneLake for cold queries.
Missing ingestion batching configuration
HIGH
Performance

Event ingestion is configured for immediate flush (MaximumBatchingTimeSpan=00:00:00). This causes excessive small file writes and high CPU usage.

→ Set batching: MaximumBatchingTimeSpan=00:00:30, MaximumNumberOfItems=10000. This reduces write amplification by ~80%.
Data Activator rules lack deduplication
HIGH
Reliability

Alert rules fire on every qualifying event, producing duplicate notifications. During a sensor failure, users received 847 alerts in 10 minutes.

→ Add suppression windows: minimum 15-minute cooldown between alerts for the same rule+entity combination.
No materialized views for common aggregations
MEDIUM
Performance

Dashboard queries aggregate raw events by minute/hour every time. Same computation repeated thousands of times daily.

→ Create materialized views for 1-min and 1-hour aggregations: .create materialized-view HourlyAgg on table RawTelemetry { ... }
No disaster recovery for Eventhouse
MEDIUM
Reliability

Single-region deployment with no follower database or backup strategy. A region outage would cause complete data loss for in-flight events.

→ Configure follower database in a secondary region. Set up continuous export to OneLake as additional backup.
Dashboard queries not optimized
LOW
Performance

3 of 5 dashboard tiles use full table scans without time filters. Query latency averages 8.2 seconds.

→ Add time range filters (where Timestamp > ago(1h)) to all dashboard queries. Use query hints for better partition pruning.
